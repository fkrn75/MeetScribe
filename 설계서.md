# MeetScribe 설계서 (Architecture)

> **프로젝트명**: **MeetScribe** (확정)
> **타깃 GPU**: RTX 3070 (VRAM 8GB)
> **작성일**: 2026-06-17
> **상태**: 설계 단계 (제작 전)
> **한 줄 정의**: 회의 녹음 파일을 넣으면 **화자별로 분리된 회의록**을 **완전 로컬**로 생성하는 Windows 데스크탑 앱

---

## 1. 개요

### 1.1 목표
- **완전 로컬·오프라인** 처리 (데이터 외부 전송 0, 기밀 회의 안전)
- **한국어 회의 우선** (large-v3 + 한국어 정렬 모델)
- **화자별 회의록** 생성 + 편집 + 다양한 형식 내보내기
- 개인 포트폴리오 + 실사용 도구

### 1.2 비목표 (MVP 범위 밖, v2 이후)
- 실시간 라이브 전사 (마이크 스트리밍)
- 화자 목소리 등록 → 자동 신원 식별
- 클라우드 동기화·협업·웹 공유
- 다국어 동시 처리 (한국어 우선, 타 언어는 동작하되 비최적)

### 1.3 핵심 설계 원칙
1. **백엔드/프론트 완전 분리** — Python은 음성 처리만, Tauri는 UI만. 성능은 100% 백엔드가 결정.
2. **장시간 작업 우선 설계** — 1시간 회의 = 수 분~수십 분. 진행률·취소·비차단(non-blocking)이 1급 요구사항.
3. **GPU/CPU 자동 분기** — 사용자 PC 사양과 무관하게 동작, GPU면 빠르게.
4. **모델 비동봉** — 첫 실행 시 다운로드(설치 파일 경량화).

## 2. 기술 스택 (확정)

| 레이어 | 기술 | 비고 |
|--------|------|------|
| 데스크탑 셸 | **Tauri 2** (Rust) | 경량, 보안, 사이드카 지원 |
| 프론트 UI | 웹 기술 (**React** 권장 / Svelte 가능) | RefBoard 웹 경험 재활용 |
| 백엔드 | **Python 3.10+ 사이드카** | PyInstaller로 단일 바이너리 번들 |
| STT | **faster-whisper** (large-v3) | CTranslate2, Whisper 대비 4배·VRAM 절반 |
| 단어 정렬 | **WhisperX align** | 한국어: `kresnik/wav2vec2-large-xlsr-korean` (기본 내장 ✅) |
| 화자 분리 | **pyannote.audio 3.1** | MIT 라이선스 ✅, HF 토큰 필요 |
| 프로세스 통신 | **로컬 HTTP API** (FastAPI + uvicorn) + **SSE** | 진행률 실시간 스트리밍 |
| 오디오 전처리 | **ffmpeg** | 포맷 변환·16kHz mono 리샘플 |

> WhisperX는 위 STT+정렬+화자분리를 한 패키지로 묶어주므로, 백엔드는 사실상 WhisperX 래핑 + API 서버화입니다.

## 3. 전체 아키텍처

### 3.1 프로세스 구조

```
┌─────────────────────────────────────────────┐
│  MeetScribe.exe (Tauri 앱)                    │
│  ┌──────────────┐      ┌───────────────────┐ │
│  │ Rust 코어     │ spawn│  Python 사이드카   │ │
│  │ (셸/lifecycle)│─────▶│  FastAPI 서버      │ │
│  │              │ kill  │  localhost:PORT   │ │
│  └──────┬───────┘      └─────────┬─────────┘ │
│         │ IPC                    │ WhisperX   │
│  ┌──────▼───────┐               │ 모델       │
│  │ WebView (UI)  │◀── HTTP/SSE ──┘            │
│  │ React         │                            │
│  └──────────────┘                            │
└─────────────────────────────────────────────┘
        모두 같은 PC, localhost 통신 (외부 전송 0)
```

### 3.2 통신 방식: 로컬 API 서버 패턴

CLI 호출 방식이 아닌 **로컬 HTTP API 서버 방식**을 택함. 이유:
- 장시간 작업의 **진행률을 SSE로 실시간** 푸시 가능
- **작업 취소**, 다중 작업 큐 관리 용이
- 프론트(웹 기술)와 자연스러운 fetch/EventSource 통신

참고 패턴: `dieharders/example-tauri-v2-python-server-sidecar`

### 3.3 데이터 흐름

```
1. 파일 드롭          → Tauri가 로컬 파일 경로 획득 (업로드 X)
2. POST /jobs         → {file_path, options} 전송, job_id 수신
3. GET /jobs/{id}/events (SSE) → {stage, percent} 실시간 수신 → 진행바
4. 완료 → GET /jobs/{id}/result → segments JSON 수신
5. 결과 뷰 렌더 + 화자/텍스트 편집 (프론트 상태)
6. 내보내기 → POST /export 또는 프론트에서 직접 생성
```

> 핵심: 파일은 같은 PC에 있으므로 **경로만 전달**, 업로드/복사 없음 → 빠르고 안전.

### 3.4 사이드카 Lifecycle (Rust 담당)

| 시점 | 동작 |
|------|------|
| 앱 시작 | 사이드카 spawn → 빈 포트 할당 → `/health` 폴링 → 준비되면 UI 활성화 |
| 작업 중 | UI는 HTTP/SSE로 통신, Rust는 감시만 |
| 앱 종료 | 사이드카 프로세스 **확실히 kill** (좀비 프로세스 방지) |
| 사이드카 크래시 | Rust가 감지 → 재시작 또는 에러 표시 |

## 4. 백엔드 설계 (Python 사이드카)

### 4.1 처리 파이프라인

```
audio 파일
  │
  ▼ [1] 전처리 (ffmpeg): 임의 포맷 → 16kHz mono WAV
  ▼ [2] STT (faster-whisper large-v3, language=ko): 세그먼트 + 단어
  ▼ [3] 정렬 (wav2vec2 ko): 단어별 정확한 타임스탬프
  ▼ [4] 화자분리 (pyannote 3.1): 화자별 발화 구간
  ▼ [5] 병합 (merge): 단어 ↔ 화자 매칭 → 최종 세그먼트
  │
  ▼ segments JSON
```

### 4.2 모듈 구조

```
sidecar/
├─ server/
│   ├─ app.py          # FastAPI 진입점, 라우팅
│   ├─ jobs.py         # 작업 큐·상태·진행률 관리
│   └─ events.py       # SSE 스트림
├─ core/
│   ├─ preprocess.py   # ffmpeg 변환·리샘플
│   ├─ transcribe.py   # faster-whisper 래핑
│   ├─ align.py        # WhisperX 단어 정렬 (ko 모델)
│   ├─ diarize.py      # pyannote 화자 분리
│   ├─ merge.py        # 화자↔단어 병합
│   └─ export.py       # txt/srt/docx/json 생성
├─ models.py           # 모델 로드/언로드/캐시, GPU 감지
├─ settings.py         # 설정 영속화
└─ main.py             # PyInstaller 진입점 (uvicorn 기동)
```

### 4.3 API 엔드포인트

| 메서드 | 경로 | 역할 |
|--------|------|------|
| GET | `/health` | 서버 준비 확인 (lifecycle용) |
| GET | `/system` | GPU 유무, 모델 다운로드 상태, 버전 |
| POST | `/jobs` | 작업 생성 `{file_path, model, language, min/max_speakers}` → `job_id` |
| GET | `/jobs/{id}` | 작업 상태 조회 |
| GET | `/jobs/{id}/events` | **SSE 진행률 스트림** `{stage, percent, message}` |
| GET | `/jobs/{id}/result` | 완료된 세그먼트 JSON |
| POST | `/jobs/{id}/cancel` | 작업 취소 |
| POST | `/export` | `{segments, format}` → 파일 생성 |
| GET/PUT | `/settings` | 모델·GPU·HF토큰·경로 설정 |
| POST | `/models/download` | 모델 다운로드 트리거 + 진행률 |

### 4.4 진행률 단계 가중치

| 단계 | 누적 % | 비고 |
|------|--------|------|
| 전처리 | ~5% | ffmpeg, 빠름 |
| STT | ~55% | 가장 오래 걸림 |
| 정렬 | ~65% | 중간 |
| 화자분리 | ~90% | 두 번째로 오래 |
| 병합 | 100% | 빠름 |

### 4.5 GPU / CPU 자동 분기

```
torch.cuda.is_available()
  ├─ True  → device="cuda", compute_type="float16", 모델 large-v3
  └─ False → device="cpu",  compute_type="int8",   모델 medium 권장 + 경고 표시
```
- 설정에서 수동 오버라이드 가능
- CPU 모드는 처리 시간이 길다는 안내 + 진행률 필수

#### 타깃 환경: RTX 3070 (VRAM 8GB) — 확정
- **large-v3를 GPU(float16)로 그대로 사용** → CPU 폴백 불필요 (정확도 최상 유지)
- VRAM 8GB는 STT+화자분리 **동시 로드 시 빠듯** → **순차 로드/언로드** 기본 적용 (STT 완료 후 `gc`+`torch.cuda.empty_cache()` → pyannote 로드)
- OOM 안전장치: `compute_type="int8_float16"` 전환 + `batch_size` 16→8 하향
- 예상 속도: 1시간 회의 ≈ **5~10분** (STT+정렬+화자분리 합산) — 실용적

### 4.6 모델 관리

| 모델 | 크기(약) | 출처 | 게이트 |
|------|---------|------|--------|
| faster-whisper large-v3 | ~3GB | HF (Systran) | ❌ |
| wav2vec2 ko (정렬) | ~1.2GB | HF (kresnik) | ❌ |
| pyannote 3.1 | ~수십MB | HF (pyannote) | ✅ **사용 동의+토큰 필요** |

- 첫 실행 시 다운로드 → 캐시 경로(`%LOCALAPPDATA%/MeetScribe/models`)
- pyannote는 앱 내 **HF 토큰 입력 UI** 제공 (설정)

## 7. 패키징 / 배포

| 항목 | 방식 |
|------|------|

## 10. 장시간(1h+) 안정성 적대검증 결과 (2026-06-17)

> 검증 방식: 3인 적대검증(Breaker 공격 / Defender 방어 / Judge 판정) + 코덱스 교차 + pyannote GitHub issues 1차 출처 재확인. **핵심 주장은 독립 경로 3개가 동일 결론**(신뢰도 최상).

### 🔴 P0 — pyannote 런타임 메모리 OOM (장시간 급소)
- **오인 정정**: §8의 "pyannote ~수십MB"는 **체크포인트 파일 크기**일 뿐 **런타임 메모리가 아님**. 이 오인이 P0의 뿌리.
- **실측(출처)**: 3h·23화자 → **18GB+ RAM**(#962), 4h embeddings → ~12GB, 클러스터링 **O(n²)** 공간복잡도, 3.1은 3.0의 **2배(6→14GB)**(#1580), 4.0은 3.3의 **VRAM 6배**(#1963).
- **2겹 방어 모두 불충분**: 설계의 유일한 OOM 완화책(int8/batch)은 **faster-whisper STT 전용** → OOM이 실제 터지는 **pyannote/정렬 단계를 못 건드림**(단계 불일치). 순차 로드도 8GB<14GB로 **양적 부족**.
- **평결**: NFR3·NFR5는 **1h 성립 / 2~3h 붕괴**. 통짜 1패스는 완주 불가 + 복구경로 없음.

### 🟠 P1
- **정렬(wav2vec2) 장시간 메모리** — WhisperX #311(극단 사례 60GB 보고)
- **취소/크래시 시 자원·임시파일 회수 미설계** (Judge는 P0 승격 권고)
- **진행률 단계 내부 공백** — STT/화자분리는 세그먼트 progress 콜백 미제공 → 3h서 진행바 수십 분 정지(체감 멈춤)
- **SSE 재연결 시 이벤트 dedup 미설계**

### ✅ 처방 4종 (MVP 필수 승격)
| 처방 | 해결 |
|------|------|
| **① 청크 단위 diarization** | 긴 오디오를 분할 처리 → O(n²)·메모리 폭발 차단 (P0 직접 해소) |
| **② 프로세스 격리** | STT/정렬/화자분리를 별 프로세스로 → OOM이 앱 전체를 안 죽임 |
| **③ 단계별 체크포인트** | 부분 실패 시 재개 → 완주 보장 |
| **④ 정렬 가드** | 정렬 실패 시 coarse 타임스탬프로 graceful degrade |

→ **NFR5 재기술 필요**: "1h 무크래시"에 더해 "2~3h는 청크 처리로 완주 보장".
